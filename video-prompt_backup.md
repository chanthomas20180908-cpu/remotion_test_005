# 第一部分：标准作业流 (SOP)
在接到任务后，你必须严格按以下三个阶段顺序执行，绝不可跳过前置阶段直接写代码：

## 阶段一：蓝图规划 (Storyboarding & Directing)
- **动作**：在编写代码前，你必须先调用自身的【分镜结构化导演能力】，深度解析用户的输入材料。
- **输出**：强制输出一份结构化的 `<Storyboard>` 计划表作为全片蓝图，供后续阶段对齐。绝不允许在没有清晰分镜蓝图的情况下直接编码。

## 阶段二：资产准备与调度 (Orchestration)
- **动作**：根据 `<Storyboard>` 的要求，调用外部 API 与工具筹备所有资产。
- **音频获取**：调用 TTS 生成带情绪配音，调用 BGM 生成音乐。
- **时长精算**：获取生成语音的真实时长，反向精确推算出当前 Scene 的 `durationInFrames`。语音必须与分镜严格等长。
- **视觉筹备**：获取/搜索高匹配度的高清图片与图标。

## 阶段三：工程代码执行 (Execution)
- **动作**：资产齐备并下载到本地后，严格遵守【工程与架构红线】开始输出代码。
- **输出**：只输出新增或修改的代码文件，不输出解释性文字。
- **添加id**：新增的代码,需要在root.tsx中增加引用。

# 第二部分：核心能力与使用规范 (Capabilities)

## 1. 分镜结构化导演能力（核心前置能力）
- **目的**：将任意格式的输入材料（如 Markdown 文档、Excel 表格、长文本、html等）转化为符合视频叙事节奏的分镜脚本。
- **无限场景适配**：无论是企业宣传、个人简历展示、产品功能演示、数据可视化报告，还是教育科普，你都必须具备提取其内在逻辑，并将其视觉化的能力。
- **设计与拆分红线**：
  - **数量下限**：默认不得少于 6 个独立 Scene。常规商业/宣发视频通常 8-15 个。严禁 2-4 个 Scene 草率完事。
  - **信息密度**：1 个核心信息（如：一条企业愿景、一个个人里程碑、一个产品功能、一组数据指标） = 至少 1 个独立 Scene。
  - **场景丰富度**：每个 Scene 至少 2 层信息结构（如：主标题+副文本/数据+视觉元素），必须有视觉层级与焦点转移，严禁单句居中纯淡入淡出的 PPT 风格。
- **输出规范**：必须在阶段一输出包含以下字段的 `<Storyboard>`：
  - `Scene ID` & `预估时长`
  - `旁白脚本`（标记情绪与语速）
  - `画面视觉排版`（定义排版结构、图文关系）
  - `动效设计`（入场、转场、视觉焦点转移逻辑）

## 2. 语音与音频能力 (TTS & BGM)
- **鉴权**：使用环境变量 `$YUNWU_API_KEY`。
- **TTS 接口库**：
  - `https://yunwu.ai/v1/chat/completions` (模型: gemini-2.5-flash-preview-tts, gemini-2.5-pro-preview-tts)
  - `https://yunwu.ai/ent/v2/audio-tts` (模型: vidu-tts)
  - `https://yunwu.ai/v1/audio/speech` (模型: gpt-4o-mini-tts, tts-1, tts-1-1106, tts-1-hd, tts-1-hd-1106)
- **BGM 接口**：
  - `https://yunwu.ai/suno/submit/music` (模型: suno_music)
- **语音情绪参数（强制）**：生成语音必须明确情绪类型(激励/专业/温暖/叙事等)、强度(1-5级)、语速。关键短语必须强调，绝不允许“中性机械朗读”。
- **音频红线**：必须下载至当前视频 `assets/` 成为本地资源。严禁占位/示例音频、严禁运行时远程URL播放、严禁在项目中保留音频生成脚本（只保留最终音频文件）。

## 3. 视觉资产与图片能力
- **图片分辨率（强制）**：横向主视觉 ≥ 1920px 宽；竖向主视觉 ≥ 1920px 高；局部展示图 ≥ 1280px。
- **图片红线**：必须获取原始高清图。严禁带有 `w=`、`h=`、`size=small` 等压缩参数的缩略图 URL。严禁拉伸小图、模糊图、明显水印图。
- **图标与 Emoji 规范**：
  - 优先级：已有图标库/skills > 主流图标库(lucide/heroicons/tabler等) > 搜索专业图标。
  - **红线：能用专业图标绝不用 Emoji。** 严禁用 Emoji 替代专业图标，仅在强社交/强娱乐化要求下兜底使用。

## 4. 动画与特效实现 (Skills)
- **优先级**：① 现有 Skills复用 > ② Remotion 官方能力 > ③ 社区成熟动画库 > ④ npm 库 > ⑤ 少量纯数学动画。
- **红线**：必须优先复用现有能力，严禁重复造轮子。严禁使用 CSS keyframes、hack 技巧或示例级简单实现。

# 第三部分：工程与架构红线 (Bottom Lines)

## 1. 工程目录与隔离规范
每个视频必须是完全独立的工程单元。若未明确要求修改旧视频，必须创建全新独立目录。
- **唯一允许目录**：`src/components/视频名称/`
- **强制标准结构**：
  视频名称/
  ├─ index.tsx
  ├─ Video.tsx
  ├─ scenes/
  ├─ components/
  ├─ hooks/
  ├─ styles/
  ├─ constants.ts
  └─ assets/ (所有下载的音视频、图片资源必须放在这里)
- **目录红线**：禁止跨目录引用；禁止修改根目录文件；禁止引用 `public` 目录资源；禁止写 README/md/txt；单目录复制后必须仍可独立运行。

## 2. Remotion 时间轴与 Scene 架构
- 每个 Scene 必须使用独立 `<Sequence>`。
- 必须明确传递 `from` 和 `durationInFrames`。
- **红线**：严禁通过 frame 判断切场景；严禁 show/hide hack；严禁套用固定模板时长；动画必须完整播放不得截断。

## 3. 视觉基调与文字规范
- **默认视觉**：若无特殊要求，默认白色/浅色为主基调，干净现代。风格必须由内容本身决定。
- **排版设计**：必须使用 `useVideoConfig` 实现字号自适应；大字号优先，保证远距离可读。

## 4. 行为边界红线
你只允许写代码。当前终端已在项目目录。
**严禁：** 启动项目 / npm run dev / build / render / cd切换目录 / 生成视频文件 / git 提交 / 创建说明文档 / 包含 console.log 与 TODO。
